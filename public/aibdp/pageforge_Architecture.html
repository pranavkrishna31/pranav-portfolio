01
System Design

How PageForge Works

PageForge is a full-stack web application built with a hybrid recommendation architecture. It blends content understanding with behavioral signals to give users book suggestions that actually match their reading habits — not just their profile.

1
User Logs In
The user registers or logs in. Their session is stored server-side using Flask's session management. From this point, every interaction they have with the platform is silently tracked in MongoDB.
2
Interactions Are Recorded
Whenever a user views a book detail page, adds a book to their cart, writes a review, or completes a purchase — an event is logged to the user_interactions collection with the username, book ID, interaction type, and a timestamp.
3
Content Vectors Are Built
All 100 books in the database are converted into text strings combining their title, author, genre, and description. These strings are fed into a TF-IDF vectorizer to create a numerical fingerprint for each book.
4
Similarity Is Computed
Cosine similarity is computed across all book vectors, producing a 100×100 similarity matrix. Every book now has a scored relationship with every other book based on content.
5
Behavioral Weights Are Applied
For each of the user's past interactions, the system looks up similar books in the matrix and multiplies that similarity score by the interaction weight (view = 1, review = 4, cart = 5, purchase = 8). Scores accumulate across all interactions.
6
Top Recommendations Are Served
Books are sorted by their final weighted score. The top 5 are returned as personalized recommendations. If the user has no interaction history, the top-rated books act as a fallback.

Technologies Used

The Tech Stack

PageForge is deliberately lean — each technology chosen serves a clear purpose. No unnecessary complexity, every tool earns its place.

🐍
Flask (Python)
The web framework powering all routes, session management, form handling, and template rendering. Flask's lightweight nature makes it ideal for a focused recommendation system without overhead. Every page — user, admin, cart, checkout, orders — is a Flask route in app.py.
Backend · Routing · Sessions
🍃
MongoDB Atlas
A cloud-hosted NoSQL database storing all books, users, orders, reviews, interactions, and contact queries. MongoDB's flexible document model allows storing nested book data inside orders without complex joins. Accessed through PyMongo in Python.
Database · Cloud · NoSQL
🔬
Scikit-learn
The machine learning library used for two core operations: TfidfVectorizer (converts book text into numerical vectors) and cosine_similarity (measures how similar two books are based on their vectors). This is the mathematical heart of the recommendation engine.
ML · TF-IDF · Cosine Similarity
🧩
Jinja2 Templates
Flask's built-in templating engine. Python data (books, recommendations, metrics) is injected directly into HTML templates using {{ variable }} syntax. The admin dashboard, book cards, and user pages are all dynamically rendered via Jinja2 from a base template.
Frontend · Templating
⚡
JavaScript (Vanilla + Chart.js)
Handles AJAX cart operations (so adding a book doesn't reload the page), toast notifications, and the sales analytics bar chart in the admin dashboard. Chart.js renders the 30-day top sellers visualization. Admin stock and sale updates are also done via async fetch calls.
Frontend · AJAX · Charts
📦
Python-dotenv + Gunicorn
python-dotenv loads environment variables from a .env file so credentials like the MongoDB URI and Flask secret key are never hardcoded. Gunicorn is the production-grade WSGI server that serves the Flask app in deployment environments.

Recommendation Engine · Module 1

TF-IDF Vectorization

The foundation of content-based recommendation. TF-IDF converts text descriptions of books into numerical vectors that a computer can compare mathematically.

📖
What is TF-IDF?
TF-IDF stands for Term Frequency–Inverse Document Frequency. It's a statistical method that measures how important a word is to a document relative to a collection of documents.

Think of it this way: the word "the" appears in every book description — it tells us nothing useful. But the word "dystopian" or "necromancer" only appears in a few books. TF-IDF gives high scores to rare, specific words and low scores to common, generic ones.
Term Frequency
TF(t, d) = (Number of times term t appears in document d) / (Total terms in document d)
Inverse Document Frequency
IDF(t) = log( Total Documents / Documents containing term t )
Combined Score
TF-IDF(t, d) = TF(t, d) × IDF(t)
How PageForge uses it: Each book's title, author, genre, and description are concatenated into a single text string (the "corpus"). The TfidfVectorizer(stop_words='english') from Scikit-learn processes all 100 book strings simultaneously, removing common English words like "the", "is", "and" before computing scores.
# Build the text corpus — one string per book for book in all_books: text = f""" {book.get('Title', '')} {book.get('Author', '')} {book.get('Genre', '')} {book.get('Description', '')} """ corpus.append(text) # Vectorize — convert text to numbers tfidf = TfidfVectorizer(stop_words='english') tfidf_matrix = tfidf.fit_transform(corpus)
Example: A book described as "A science fiction epic about space exploration and AI consciousness" would get high TF-IDF scores for "science fiction", "space exploration", "AI", and "consciousness" — and near-zero for common words like "about" or "and". These scores form its unique numerical identity.
Output: A matrix of shape (100 books × N unique words), where each row is a book's TF-IDF vector — its mathematical fingerprint.

Recommendation Engine · Module 2

Cosine Similarity

Once every book has a TF-IDF vector, cosine similarity measures how "close" two books are in meaning — the smaller the angle between two vectors, the more similar the books.

📐
What is Cosine Similarity?
Imagine each book as an arrow pointing in a direction in a high-dimensional space — where each dimension represents a unique word. Two books pointing in almost the same direction (small angle between them) are similar. Two books pointing in completely different directions (90° apart) are unrelated.

Cosine similarity calculates the cosine of the angle between two vectors. A score of 1.0 means identical, 0.5 means moderately similar, and 0.0 means no shared themes at all.
Cosine Similarity Formula
cos(θ) = (A · B) / (|A| × |B|)
Where A · B is the dot product (how much vectors overlap) and |A| × |B| normalizes for vector length, ensuring longer descriptions don't unfairly dominate shorter ones.
# Compute pairwise similarity across all books similarity_matrix = cosine_similarity( tfidf_matrix, tfidf_matrix ) # Result: 100×100 matrix # similarity_matrix[i][j] = similarity score between book i and book j # Always 1.0 on the diagonal (a book vs itself)
Practical example: Book 12 is "1984" (dystopian political fiction). Book 47 is "Brave New World" (also dystopian). Their TF-IDF vectors both score high for words like "dystopian", "society", "control", "future" — so cosine similarity returns ~0.8 (very similar). Compare this to "Pride and Prejudice" (romance/manners) which would score ~0.05 against "1984".
Output: A 100×100 symmetric matrix. Each cell contains a score between 0 and 1 representing textual similarity between two books. This matrix is the core lookup table used during recommendation generation.

Recommendation Engine · Module 3

Content-Based Filtering

Content-based filtering recommends items based on the properties of items the user has already interacted with — not on what other users did.

🧠
What is Content-Based Filtering?
If you enjoyed a detective novel, a content-based system asks: "What other books share similar characteristics — same genre, similar themes, same author style?" It looks at the properties (content) of items, not the crowd's behavior.

In PageForge, the "content" of a book is its title + author + genre + description — all captured in the TF-IDF vector. The system finds books with the highest cosine similarity to books the user has already engaged with.
Key advantage: Works well even for new users with limited history, and doesn't require knowing what other users are reading. It's entirely based on the book's own metadata.
🔄
How It Works in PageForge
For each book the user has interacted with (viewed, carted, reviewed, purchased), the system:

1. Finds that book's index in the all_books list
2. Looks up its row in the similarity matrix
3. Gets similarity scores against every other book
4. Multiplies those scores by the interaction weight
5. Adds to a running weighted_scores dictionary

The books that accumulate the highest total weighted scores across all of the user's past interactions become the final recommendations.
Fallback behavior: If a new user has no interactions yet, the system falls back to recommending the top-5 highest-rated books — ensuring recommendations are always meaningful even on the first visit.

Recommendation Engine · Module 4

Multi-Signal Behavioral Weighting

Not all user interactions are equal. Buying a book is a much stronger signal of interest than merely glancing at it. PageForge assigns different weights to each type of interaction to make recommendations progressively smarter.

⚖️
Interaction Weight Table
Each action contributes a different weight to the recommendation score. Higher-intent actions carry more influence.
Interaction Weight Why this weight?
👁 View (Detail Page) 1 Browsing is passive — low intent signal
⭐ Review / Rating 4 Writing a review means real engagement
🛒 Cart Addition 5 Adding to cart shows purchase intent
💳 Purchase 8 The strongest possible interest signal
💡
Rating Boost for Reviews
Reviews carry an extra bonus. If a user gave a book 5 stars, the weight becomes 4 + 5 = 9. If they gave 2 stars, it becomes 4 + 2 = 6. This means highly-rated books influence recommendations more strongly than poorly-rated ones — making the system sensitive to the user's actual preferences, not just their activity.
# Extra boost: add rating value to review weight if interaction_type == "review": rating_value = interaction.get( "InteractionValue", 0 ) weight += rating_value # e.g., 4 + 5 = 9 # Multiply similarity score by behavioral weight weighted_scores[candidate_id] = ( weighted_scores.get(candidate_id, 0) + (similarity_score * weight) )
Result: A book you purchased and a book you rated 5 stars will generate much stronger "gravitational pull" on similar books in the recommendation engine than a book you merely glanced at.

Evaluation Metrics · Module 1

Precision@K

A metric that tells you: of the K books we recommended, how many were actually relevant to the user? It measures the quality of recommendations.

Precision@K
How accurate are our recommendations?
K is the number of recommendations shown (in PageForge, K = 5). Relevant books are those the user has actually purchased — we treat purchase history as ground truth for what the user truly wanted.

Precision@K asks: "Out of the 5 books we recommended, how many did the user actually end up buying or would have wanted?" A score of 100% means all 5 recommendations were spot-on. A score of 40% means 2 out of 5 were relevant.
Formula
Precision@K = (Relevant Books in Top-K) ÷ K
Worked example:

K = 5 (we show 5 recommendations)
Recommended: [Book A, Book B, Book C, Book D, Book E]
User's purchased books (relevant): [Book A, Book C, Book F, Book G]

Hits = 2 (Book A and Book C are in both lists)
Precision@5 = 2 / 5 = 0.40 → 40%

This means 40% of what we showed was genuinely useful to the user.
def precision_at_k(recommended, relevant, k): recommended_at_k = recommended[:k] hits = sum( 1 for book_id in recommended_at_k if book_id in relevant ) return hits / k if k > 0 else 0
In the admin dashboard: This value is shown as a percentage (multiplied by 100 and rounded to 2 decimals). It gives the admin a live view of how accurate the recommendation engine currently is.

Evaluation Metrics · Module 2

Recall@K

Where Precision measures accuracy, Recall measures coverage. It asks: of all the books the user would have liked, how many did we actually surface in our recommendations?

Recall@K
How much did we cover?
Recall@K measures the completeness of recommendations. Even if all 5 recommendations are perfect (Precision = 100%), if the user actually liked 20 books and we only found 5, our recall is 25%.

In PageForge, "relevant books" are all books that appear in any user's purchase history. "Recommended books" are the top 5 generated by weighted behavioral scores. Recall tells the admin how much of the universe of relevant books the engine is successfully discovering.
Formula
Recall@K = (Relevant Books in Top-K) ÷ Total Relevant Books
Worked example:

K = 5 (showing 5 recommendations)
Recommended: [Book A, Book B, Book C, Book D, Book E]
All relevant books ever purchased: [Book A, Book C, Book F, Book G, Book H, Book J, Book K, Book L]

Hits = 2 (Book A and Book C)
Total relevant = 8
Recall@5 = 2 / 8 = 0.25 → 25%

We found 2 out of 8 relevant books. Good Precision, lower Recall.
def recall_at_k(recommended, relevant, k): recommended_at_k = recommended[:k] hits = sum( 1 for book_id in recommended_at_k if book_id in relevant ) return hits / len(relevant) if relevant else 0
Precision vs Recall tradeoff: There's always a tension. Showing more recommendations (larger K) usually improves Recall but can hurt Precision. PageForge uses K=5 — a small, precise recommendation set prioritizing quality over coverage.

Evaluation Metrics · Module 3

Click-Through Rate (CTR)

CTR measures how often users actually click on a book after seeing it. It's a direct measure of user engagement and the appeal of displayed content.

CTR
Are users engaging with what they see?
Every time a book appears on the user browse page, its ShowCount increments. Every time a user clicks through to the book detail page or completes a purchase, its ClickCount and DetailViews increment.

CTR divides total clicks by total impressions and multiplies by 100 to express as a percentage. A high CTR means users find a book's title, genre, and rating compelling enough to explore further.
Formula
CTR (%) = (ClickCount + DetailViews) / ShowCount × 100
Example:

"Dune" has been shown in browse results 200 times (ShowCount = 200).
It's been clicked 40 times (ClickCount = 30, DetailViews = 10).

CTR = 40 / 200 × 100 = 20%

20% of users who saw Dune clicked on it. The admin dashboard also shows Average CTR across all books as a platform-wide engagement indicator.
# Computed per book in the admin route click_count = ( book.get("ClickCount", 0) + book.get("DetailViews", 0) ) show_count = book.get("ShowCount", 1) if show_count > 0: ctr_value = (click_count / show_count) * 100 book["CTR"] = round(ctr_value, 2)
Used for: The admin dashboard shows a CTR column for every book, and the top-5 highest CTR books are tracked separately. This helps identify which books are visually or topically compelling to readers — useful for curation and marketing.

Data Layer · Behavioral Signals

How Interactions Are Tracked

Every meaningful user action silently generates a document in the user_interactions collection. These events form the behavioral dataset the recommendation engine learns from.

👁
View Interaction
Triggered on every visit to a book's detail page (/book/<id>). Simultaneously, the book's DetailViews counter increments by 1. The logged event has InteractionType = "view" with weight 1.
🛒
Cart Interaction
Triggered when a user clicks "Add to Cart". Only logged once per book (duplicate cart additions are prevented). The event has InteractionType = "cart" with weight 5 — a high-intent signal showing the user considered buying.
⭐
Review Interaction
Triggered on review submission. The rating (1–5) is stored as InteractionValue and gets added to the base weight of 4. Simultaneously, the book's dynamic RatingMean is recalculated using an incremental average formula.
💳
Purchase Interaction
Triggered on checkout completion, one event per book in the cart. The purchase price is stored as InteractionValue. Weight 8 — the strongest possible signal. Each purchased book also increments ClickCount for CTR tracking.
Dynamic Rating Update: When a user submits a review, the book's RatingMean doesn't just get re-averaged from scratch. Instead, an incremental formula is used: new_mean = (current_mean × current_count + new_rating) / (current_count + 1) This is O(1) — no need to re-fetch all previous reviews from the database.

Application Modules

User-Facing Features

The user side of PageForge is a complete e-commerce reading experience — search, discovery, recommendations, cart, checkout, orders, reviews, and support.

🔐
Auth: Register & Login
Users register with a username + password (stored in the users collection with Role: "user"). On login, Flask stores the username and role in a server-side session. Role-based access control redirects admins to the admin dashboard and users to the browse page.
🔍
Search & Filter
Users can search by title or author using MongoDB regex queries. Books can be filtered by genre (exact match from the 7 available genres) and sorted by Price (ascending) or Rating (descending). Every filtered/sorted book view increments ShowCount.
✨
Personalized Recommendations
Displayed at the top of the browse page in a dedicated "Recommended For You" section, labelled with an AI pill badge. These 5 books are generated fresh on every page load using the full TF-IDF + cosine + behavioral weighting pipeline.
🛍
Cart & Checkout
Cart IDs are stored in the Flask session (not the database). The checkout flow collects name, address, phone, and payment method. On order placement, book documents are embedded directly into the order record, purchase interactions are logged, ClickCount increments, and cart is cleared.
📦
Order History
All past orders for the logged-in user are fetched from MongoDB, sorted by date descending. Each order card shows delivery details, payment method, list of books purchased, and total paid — a full purchase receipt.
📬
Contact & Support
Users can submit queries via a contact form (username, email, subject, message). Submissions are saved to the contact_queries collection. A flash message confirms submission with a 24–48 hour response expectation.

Application Modules

Admin Analytics Dashboard

The admin dashboard is a recommendation control center showing live precision, recall, CTR, top sellers, and full inventory — all computed fresh on each page load.

📊
KPI Cards: Precision, Recall, Avg CTR
Three headline metrics displayed prominently at the top of the dashboard:

• Precision@5 — computed using the platform-wide most-interacted books as recommended set, and purchase history as relevant set.

• Recall@5 — same sets, different formula denominator (total relevant vs K).

• Average CTR — mean CTR across all 100 books in the database.
📈
30-Day Top Sellers Chart
A bar chart (rendered with Chart.js) showing the 10 most-sold book titles in the last 30 days. Orders are filtered by date (Date >= now - 30 days), titles are counted with Python's Counter, and the top 10 are passed to the template as label/value arrays.
📋
Inventory Table
A scrollable table showing every book in the database with its computed CTR, raw ClickCount, and ShowCount. This lets the admin identify high-engagement and low-engagement books at a glance. CTR is computed in-memory — not stored in the database.
🛠
Stock & Sales Management
Two AJAX-powered forms in the admin sidebar:

• Update Stock — sets StockAvailable for a book by its ID.

• Add Sale — logs a sale record, decrements stock by the sold units, and prevents negative inventory (sale is rejected if units exceed available stock).

Both use fetch() POST requests returning JSON, with page reload on success.
How recommended_books are computed for admin metrics: Unlike the user-facing recommendations (which are personalized per user), the admin uses a global recommendation set. All user interactions are aggregated using Python's Counter with weighted sums, and the top 5 book IDs by total score become the "recommended" set for computing Precision@K and Recall@K across all users.

Security & Configuration

Security Architecture

PageForge's security model is appropriate for an academic/portfolio project with room for production hardening. Here's what's in place and what's not.

🔒
What's Secured
Environment Variables: MongoDB URI and Flask SECRET_KEY are stored in a .env file (not committed to source control) and loaded at runtime via python-dotenv.

Session Management: Flask uses the SECRET_KEY to cryptographically sign session cookies, preventing tampering. Sessions hold username and role — used for all route access checks.

Role-Based Access: Every admin route checks session.get('role') != 'admin' before proceeding. Every user route checks for the user role. Unauthorized access redirects to login.

Inventory Protection: The add_sale endpoint checks available stock before accepting a sale, preventing negative inventory via code-level validation.
⚠️
Areas for Improvement (Production)
Password Hashing: Currently stored as plain text. Should use werkzeug.security.generate_password_hash and check_password_hash.

MongoDB IP Whitelisting: Atlas allows restricting connections to specific IP addresses — recommended for production to prevent unauthorized database access.

CSRF Protection: Form submissions currently lack CSRF tokens. Flask-WTF can add this protection automatically.

Input Validation: Server-side validation is basic — production would benefit from stricter sanitization of all form fields.
Deployment note: The Procfile contains web: gunicorn app:app, enabling deployment to platforms like Heroku or Render where Gunicorn serves the Flask app in a production-ready multi-worker configuration instead of Flask's built-in development server.
