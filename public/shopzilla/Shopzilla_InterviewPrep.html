What is MERN?

MERN is an acronym for four JavaScript technologies that together cover every layer of a modern web application. The key advantage is that you use a single language — JavaScript — across the entire stack, which simplifies development considerably.

🍃

🚂

Full Request Flow — Step by Step

What happens when a user opens Shopzilla
1

Browser hits the Netlify URL

Netlify serves the static React build (HTML + JS bundles). The main.js bundle is downloaded to the browser — this is Client-Side Rendering (CSR). No server renders HTML; React does it in the browser.

2

React mounts & useEffect fires API call

HomePage.js mounts. The useEffect hook triggers an Axios/fetch call to https://shopzilla31.onrender.com/api/products.

3

Express receives the request

The request hits server.js on Render. Express routes it via app.use('/api/products', productRoutes). The route handler calls productController.getProducts().

4

Cache check (NodeCache)

Controller checks the in-memory NodeCache. Cache hit → returns data immediately (sub-millisecond). Cache miss → proceeds to query MongoDB Atlas.

5

Mongoose queries MongoDB Atlas

Mongoose runs: Product.find().select('name price image').limit(10).lean(). This hits Cluster0, database "test", collection "products". MongoDB returns BSON which Mongoose converts to JSON.

6

Cache store & JSON response

Data is stored in NodeCache with a 60-second TTL. Express sends res.json(products) with HTTP 200 back to the React frontend.

7

React updates UI with useState

setProducts(data) updates React state. React re-renders the ProductCard components. The product grid appears in the browser. Average response: ~245ms over 200 Postman iterations.

Key Point

CSR (Client-Side Rendering) means the server sends an empty HTML shell + JS bundle. React renders the content in the browser. This is different from SSR (Server-Side Rendering, like Next.js) where the server sends fully-rendered HTML. CSR is faster at navigation but slower at first meaningful paint for SEO purposes.

MongoDB & Mongoose

MongoDB is the NoSQL document database powering Shopzilla's data layer. Mongoose is the ODM (Object Data Modelling) library that adds structure, validation, and a clean query API on top of raw MongoDB.

What is MongoDB?

MongoDB stores data as BSON documents (Binary JSON) inside collections. Unlike SQL, there are no rigid table schemas — each document can have different fields. This flexibility makes it ideal for e-commerce where products might have varied attributes.

📚
Collection
Equivalent to a SQL table. Shopzilla has: products, users, orders, contacts collections inside the test database on Cluster0.
📄
Document
A single record — like a row in SQL. Stored as JSON-like BSON. Each document gets a unique _id (ObjectId) automatically assigned by MongoDB.
🔑
ObjectId
A 12-byte BSON type MongoDB auto-generates as the primary key. Example: ObjectId("507f1f77bcf86cd799439011"). Encodes timestamp, machine, process, and a counter.

MongoDB Atlas — Cloud Setup

MongoDB Atlas is the cloud-hosted version of MongoDB. Instead of running MongoDB locally, Atlas manages the server, backups, scaling, and monitoring for you. Your Shopzilla uses Cluster0 with database name test.

What is Mongoose?

Mongoose is an ODM — Object Data Modelling library. It sits between your Express code and MongoDB, giving you Schemas (structure), Models (query interface), Validation, Virtuals, and Middleware (hooks). Without Mongoose, you'd write raw MongoDB driver queries which are verbose and unstructured.

🔧 Schema

Defines the shape of documents — what fields exist, their types, whether they're required, default values, etc. This is compile-time structure on top of MongoDB's schema-less nature.

🏭 Model

A Model is compiled from a Schema. It's the constructor/class you use to create, read, update, delete documents. Example: Product.find(), User.findById(id).

Key Mongoose Query Methods used in Shopzilla

MethodWhat it doesShopzilla Usage
.find()Returns all matching documentsGet all products
.findById(id)Finds document by _idGet single product page
.findOne(query)Returns first matching documentCheck if email exists at login
.select('f1 f2')Return only specified fields (projection)select('name price image') — avoids fetching description etc.

Why .lean() matters

By default, Mongoose wraps query results in Mongoose Document objects which have methods like .save(), .populate(), getters, setters, etc. These add memory and CPU overhead. .lean() tells Mongoose to skip this wrapping and return plain JavaScript objects directly — making reads significantly faster, especially for large collections. Use it when you only need to read and send data, not modify it.

Referencing vs Embedding

In MongoDB, you can either embed related data (put it inside the same document) or reference it using an ObjectId. Shopzilla references User in Order using ObjectId, ref: 'User'. This allows using .populate('user') to automatically fetch the user details when querying orders — similar to a SQL JOIN.

Node.js & Express.js

Node.js is the JavaScript runtime that powers your backend server. Express.js is the web framework layered on top of it that handles routes, middleware, and the request-response cycle.

How Node.js Works

Node.js runs on Google's V8 engine (same engine in Chrome). It's single-threaded but uses an event loop for non-blocking I/O. This means when your code waits for a database query, Node doesn't sit idle — it moves on to handle other requests and comes back when the DB responds. This is why Node is highly performant for API servers with many concurrent connections.

Key Concept — Event Loop

When Node.js executes await Product.find(), it hands off the DB query to the OS, moves to other work, and resumes execution when the DB responds. This is the non-blocking I/O model — it's why Node can handle thousands of concurrent connections on a single thread.

The server.js Entry Point

server.js — App Bootstrap JavaScript 
const express = require('express');
const connectDB  = require('./config/db');
const cors = require('cors');

const productRoutes = require('./routes/productRoutes');
const userRoutes    = require('./routes/userRoutes');
const orderRoutes   = require('./routes/orderRoutes');

connectDB(); // Connect to MongoDB Atlas

const app = express();

// ── MIDDLEWARE ──────────────────────────────
app.use(cors());                         // Allow cross-origin requests (Netlify → Render)
app.use(express.json());               // Parse JSON request bodies
app.use(express.urlencoded({ extended: true }));

// ── ROUTES ──────────────────────────────────
app.use('/api/products', productRoutes);
app.use('/api/users',    userRoutes);
app.use('/api/orders',   orderRoutes);

// ── START SERVER ─────────────────────────────
const PORT = process.env.PORT || 5000;
app.listen(PORT, () => console.log(`Server running on port ${PORT}`));

Middleware — The Request Pipeline

In Express, middleware functions are functions that have access to the request object (req), the response object (res), and the next middleware function (next). Every incoming request passes through the middleware chain in order before reaching the route handler.

🌐
cors()
Cross-Origin Resource Sharing. Without this, the browser blocks requests from Netlify (port 3000 or HTTPS) to your Render backend (different domain). CORS middleware adds the necessary Access-Control-Allow-Origin headers.
📝
express.json()
Parses incoming request bodies with Content-Type application/json. Without this, req.body would be undefined when the frontend sends login credentials or order data as JSON.
🔐
authMiddleware.js
Custom middleware. Reads the JWT token from req.headers.authorization, verifies it with jwt.verify(). If valid, attaches the decoded user to req.user and calls next(). If invalid, sends 401 Unauthorized.

REST APIs & Routes

REST (Representational State Transfer) is an architectural style for APIs. Shopzilla's backend exposes RESTful endpoints that the React frontend calls using HTTP methods (GET, POST, PUT, DELETE).

What Makes an API RESTful?

Stateless
Each request contains all the information needed. The server doesn't remember previous requests. This is why JWT is sent with every protected request — the server doesn't store session state.
Resource-Based URLs
URLs represent resources (nouns), not actions (verbs). Use /api/products, not /getProducts. The HTTP method (GET/POST) tells you the action.
HTTP Methods = CRUD
GET = Read, POST = Create, PUT/PATCH = Update, DELETE = Delete. This mapping is fundamental to REST architecture.

Route & Controller Code Pattern

routes/productRoutes.js JavaScript 
const express = require('express');
const router  = express.Router();
const { getProducts, getProductById } = require('../controllers/productController');

router.get('/',    getProducts);      // GET /api/products
router.get('/:id', getProductById); // GET /api/products/:id

controllers/productController.js JavaScript 
const Product  = require('../models/Product');
const NodeCache = require('node-cache');
const cache = new NodeCache({ stdTTL: 60 }); // 60-second TTL

const getProducts = async (req, res) => {
  try {
    // 1. Check cache first
    const cached = cache.get('products');
    if (cached) {
      return res.json(cached); // Cache HIT → instant response
    }

    // 2. Cache MISS → query MongoDB
    const products = await Product
      .find({})
      .select('name price image countInStock') // projection
      .limit(10)                               // pagination cap
      .lean();                                  // plain JS objects

    // 3. Store in cache
    cache.set('products', products);

    res.json(products);
  } catch (error) {
    res.status(500).json({ message: error.message });
  }
};

JWT & bcrypt Authentication

Shopzilla uses two security libraries together: bcryptjs to safely hash passwords before storing them, and jsonwebtoken (JWT) to authenticate users on subsequent requests without storing session state on the server.

Password Hashing with bcrypt

You never store plain-text passwords. bcryptjs uses the Blowfish cipher algorithm to create a one-way hash. Even if your database is compromised, attackers get hashes — not passwords. The hash includes a randomly generated "salt" which means the same password hashes differently each time, preventing rainbow table attacks.

🧂 What is a Salt?

A salt is a random string added to the password before hashing. This ensures two users with the same password get different hashes. The salt is automatically embedded in the bcrypt hash output itself.

🔁 Salt Rounds

The "cost factor" or "rounds" (typically 10) determines how many times the algorithm iterates. Higher rounds = slower hashing = harder to brute force. 10 rounds is the industry standard balance between security and speed.

controllers/userController.js — Registration JavaScript 
const bcrypt = require('bcryptjs');
const jwt    = require('jsonwebtoken');
const User   = require('../models/User');

const registerUser = async (req, res) => {
  const { name, email, password } = req.body;

  // 1. Check if user already exists
  const exists = await User.findOne({ email });
  if (exists) return res.status(400).json({ message: 'User already exists' });

  // 2. Hash the password (10 salt rounds)
  const salt           = await bcrypt.genSalt(10);
  const hashedPassword = await bcrypt.hash(password, salt);

  // 3. Save user with hashed password
  const user = await User.create({ name, email, password: hashedPassword });

  // 4. Generate JWT token
  const token = jwt.sign(
    { id: user._id },                   // payload
    process.env.JWT_SECRET,            // secret key from .env
    { expiresIn: '30d' }              // token expires in 30 days
  );

  res.status(201).json({ _id: user._id, name, email, token });
};
Login — bcrypt comparison JavaScript 
const loginUser = async (req, res) => {
  const { email, password } = req.body;

  // 1. Find user by email
  const user = await User.findOne({ email });

  // 2. Compare entered password with stored hash
  if (user && await bcrypt.compare(password, user.password)) {
    // bcrypt.compare() hashes the input and compares — never decrypts the stored hash

    const token = jwt.sign({ id: user._id }, process.env.JWT_SECRET, { expiresIn: '30d' });
    res.json({ _id: user._id, name: user.name, email, token });
  } else {
    res.status(401).json({ message: 'Invalid email or password' });
  }
};

What is a JWT?

JSON Web Token is a compact, URL-safe token format used to securely transmit information between parties as a JSON object. A JWT has three parts separated by dots: Header.Payload.Signature.

middleware/authMiddleware.js — Protecting Routes JavaScript 
const jwt  = require('jsonwebtoken');
const User = require('../models/User');

const protect = async (req, res, next) => {
  let token;

  // Frontend sends: Authorization: Bearer eyJhbGci...
  if (req.headers.authorization?.startsWith('Bearer')) {
    token = req.headers.authorization.split(' ')[1];

    const decoded = jwt.verify(token, process.env.JWT_SECRET);
    // decoded = { id: '60767d78...', iat: ..., exp: ... }

    req.user = await User.findById(decoded.id).select('-password');
    // Attach user to req (minus password) for use in route handlers

    next(); // Proceed to the actual route handler
  } else {
    res.status(401).json({ message: 'Not authorized, no token' });
  }
};
Complete Authentication Flow
1

User submits email + password

React's LoginPage.js sends POST to /api/users/login with JSON body via Axios.

2

Backend verifies with bcrypt

bcrypt.compare(enteredPassword, storedHash) — hashes the input and checks against the stored hash. This is a CPU-intensive operation, explaining the ~895ms average login time.

3

JWT token generated & sent

jwt.sign({id: user._id}, JWT_SECRET, {expiresIn: '30d'}) creates a signed token. Sent back in the JSON response body.

4

Frontend stores token

React stores the JWT in localStorage (or Context state). On subsequent requests to protected endpoints, the token is sent in the Authorization: Bearer <token> header.

5

authMiddleware verifies token

jwt.verify(token, JWT_SECRET) validates the signature. If valid, req.user is populated and next() is called to proceed to the route handler.

NodeCache & Performance

Caching is the practice of storing frequently-accessed data in a fast-access layer (memory) so that repeated requests don't hit the database every time. Shopzilla uses NodeCache — an in-process, in-memory cache for Node.js.

How NodeCache Works

NodeCache stores key-value pairs directly in Node.js's process memory (heap). When the controller receives a request, it first checks the cache. If the key exists and hasn't expired (TTL), it returns the cached value. Otherwise it queries MongoDB, caches the result, and returns it.

NodeCache Setup & Usage JavaScript 
const NodeCache = require('node-cache');

// stdTTL = standard Time-To-Live in seconds
// checkperiod = how often (seconds) NodeCache checks for expired keys
const cache = new NodeCache({ stdTTL: 60, checkperiod: 120 });

// SET: Store data in cache
cache.set('products', products); // key: 'products', value: array, TTL: 60s

// GET: Retrieve from cache (returns undefined if expired/not found)
const cached = cache.get('products');

// DEL: Invalidate cache (when product is updated)
cache.del('products');

// FLUSH: Clear all cache
cache.flushAll();
Cache Hit vs Cache Miss Flow
→

Request arrives at GET /api/products

Controller executes. First operation: cache.get('products')

✅

CACHE HIT — data found & not expired

Returns cached array immediately. No DB query. Response time: sub-millisecond from Node process memory. This is the fast path for the ~245ms average you measured.

❌

CACHE MISS — key expired or first request

Falls through to MongoDB query. After data is fetched, it's stored with cache.set('products', data). Subsequent requests within 60 seconds hit the cache instead.

Query Optimisation Techniques

TechniqueMethodWhy it matters
.select()Product.find().select('name price image')Projection — MongoDB only reads specified fields from disk. Less data = faster query + smaller network response payload.
.limit().limit(10)Tells MongoDB to stop after 10 documents. Without limit, a full collection scan returns every document — slow on large collections.
.lean().lean()Skips Mongoose Document wrapping. Returns plain JS objects. Reduces memory allocation and CPU overhead for read-heavy endpoints.
Lazy Loading<img loading="lazy">Browser defers loading off-screen images. Only loads images as user scrolls. Reduces initial page network payload significantly.
NodeCache vs Redis

NodeCache is in-process — it lives in the same Node.js memory. This means: (1) it resets when the server restarts, (2) it can't be shared across multiple server instances (can't scale horizontally). Redis is a separate in-memory data store accessed over a network — it persists across restarts, can be shared by multiple server instances, and supports advanced data structures. For production at scale, Redis is the right choice. NodeCache is appropriate for simple single-instance deployments like Shopzilla on Render.

Core React Concepts in Shopzilla

⚡

Hook for side effects — API calls, subscriptions, timers. Runs after render. Used in Shopzilla to fetch products from the backend when a page mounts.

The Virtual DOM — Why React is Fast

When state changes, React doesn't update the actual DOM directly (which is slow). Instead it creates a Virtual DOM — a lightweight in-memory copy of the DOM tree — computes the diff (reconciliation), and only applies the minimal set of changes to the real DOM. This is why React can handle frequent updates efficiently.

HomePage.js — useEffect + useState Pattern JavaScript 
import React, { useState, useEffect } from 'react';
import axios from 'axios';
import ProductCard from '../components/ProductCard';

const HomePage = () => {
  const [products, setProducts] = useState([]); // initial: empty array
  const [loading, setLoading]   = useState(true);

  useEffect(() => {
    // Runs once on component mount ([] dependency array)
    const fetchProducts = async () => {
      const { data } = await axios.get(
        `https://shopzilla31.onrender.com/api/products`
      );
      setProducts(data);   // triggers re-render with product data
      setLoading(false);
    };
    fetchProducts();
  }, []); // [] = run only once on mount

  return (
    <div>
      {loading ? <Loader /> : products.map(p => (
        <ProductCard key={p._id} product={p} /> // key = MongoDB _id
      ))}
    </div>
  );
};

State & Context API

State management is how React shares and updates data across components. Shopzilla uses React's built-in Context API to manage global state like cart items and user login — avoiding "prop drilling".

What is Prop Drilling?

If you have data in a parent component that a deeply nested child needs, you'd have to pass it through every intermediate component as a prop — even if those middle components don't need it. This is "prop drilling" and becomes unmanageable at scale. Context API solves this by creating a global store any component can subscribe to directly.

context/CartContext.js JavaScript 
import React, { createContext, useContext, useState } from 'react';

// 1. Create the context
const CartContext = createContext();

// 2. Provider component — wraps the whole app in App.js
export const CartProvider = ({ children }) => {
  const [cartItems, setCartItems] = useState([]);

  const addToCart = (product) => {
    setCartItems(prev => [...prev, product]);
  };

  const removeFromCart = (id) => {
    setCartItems(prev => prev.filter(item => item._id !== id));
  };

  return (
    <CartContext.Provider value={{ cartItems, addToCart, removeFromCart }}>
      {children}   // All child components can access cartItems, addToCart, removeFromCart
    </CartContext.Provider>
  );
};

// 3. Custom hook — clean way to consume context
export const useCart = () => useContext(CartContext);

// Usage in any component:
// const { cartItems, addToCart } = useCart();
Context API vs Redux

Context API is React's built-in global state solution — great for small-to-medium apps like Shopzilla. Redux is an external library offering a more structured flux pattern with actions, reducers, and a single store — better for very large apps with complex state. Context + useReducer can approximate Redux patterns without the extra dependency.

Why .env files are never committed to Git

The .env file contains secrets like your MongoDB connection string (with username/password) and JWT secret key. Key Q&A

Common questions about this project with detailed technical answers.

🔴 "Explain your project in 2 minutes."

Shopzilla is a full-stack e-commerce app I built with the MERN stack — MongoDB, Express, React, and Node.js. The frontend is deployed on Netlify, the backend on Render, and the database on MongoDB Atlas. Users can browse products, register, log in, add items to cart, and place orders. I implemented JWT authentication with bcrypt password hashing for security, and NodeCache for performance — which brought the product API response time down to ~245ms over 200 Postman iterations. The Lighthouse score is 76 with near-perfect FCP and LCP times.

🔴 "Why MongoDB over SQL for this project?"

E-commerce products can have highly varied attributes — a phone has storage and RAM, a shirt has size and colour. A flexible document schema fits this better than rigid SQL tables. MongoDB also stores data as JSON-like BSON which aligns naturally with the JavaScript/Node.js stack, reducing data transformation overhead. That said, for complex relational queries like order-to-user-to-product joins, a relational DB would be more natural — which is a known trade-off.

🔴 "Why is your login API slower than your product API?"

bcrypt's compare operation is intentionally CPU-intensive — running 10 salt rounds makes it computationally expensive for brute-force attackers. This is by design. The product API is fast because it benefits from NodeCache (in-memory hit) and optimised Mongoose queries using .select(), .limit(), and .lean(). These are fundamentally different workloads.

🔴 "How does JWT authentication work?"

JWT is a stateless authentication mechanism. On login, the server generates a signed token containing the user's ID and an expiry time. This token is sent to the frontend, which stores it and attaches it to every protected request in the Authorization header as "Bearer token". The server's authMiddleware verifies the signature using the JWT_SECRET — if valid, the user is authenticated. No session is stored on the server, which makes it horizontally scalable.

🔴 "What is .lean() and why did you use it?"

By default, Mongoose wraps query results in Mongoose Document objects which carry methods, getters, setters, and other overhead. .lean() tells Mongoose to return plain JavaScript objects instead — skipping all that wrapping. For read-only endpoints like GET /api/products where we're just sending data to the client, this is faster in both memory usage and CPU time. The trade-off is you lose Mongoose Document methods like .save() — but for reads, that's perfectly fine.

🔴 "What's the difference between CSR and SSR?"

CSR (Client-Side Rendering) is what Shopzilla uses — Netlify sends an empty HTML shell with a JavaScript bundle. React then runs in the browser and renders the UI. SSR (Server-Side Rendering), like Next.js, renders the HTML on the server before sending it. SSR is better for SEO and initial load speed but adds server complexity. CSR is simpler to deploy and works well for app-like interfaces where SEO is less critical.

🔴 "What would you improve in Shopzilla?"

Several things: (1) Replace NodeCache with Redis for distributed caching that persists across server restarts and scales horizontally. (2) Add MongoDB indexes on frequently queried fields like price and category for faster queries. (3) Migrate to Next.js for SSR to improve SEO and initial load. (4) Integrate a payment gateway like Razorpay. (5) Implement proper pagination instead of a basic limit. (6) Fix the CLS score (0.712) by giving images explicit width and height attributes.
