STEP 01

Data Loading & Preprocessing

Why the raw data pipeline matters — preparing structured, normalized, and augmented inputs for the deep learning models.

Autonomous vehicles depend on diverse sensor data captured in wildly varying real-world conditions — different weather, lighting, road geometries, and object angles. Feeding inconsistent raw images directly into a neural network would make training unstable and the learned model brittle. The preprocessing pipeline solves this by standardising every input before the model ever sees it.

Datasets Used

GTSRB
German Traffic Sign Recognition Benchmark — 43 classes of traffic signs used for the classification task. Provides labelled images of speed limits, stop signs, warnings, and more. This is the primary dataset for the CNN classifier.
43 ClassesRGB ImagesClassification
KITTI
Multi-sensor driving dataset from Karlsruhe, Germany. Contains RGB camera images, LiDAR 3D point clouds, GPS/IMU data, and fine-grained annotations for cars, pedestrians, and cyclists with both 2D and 3D bounding boxes. Used for object detection with YOLOv8.
RGB CameraLiDARGPS+IMU2D/3D BBoxes
nuScenes
Full 360° perception dataset from Motional with 6 cameras, LiDAR, Radar, and GPS/IMU. Contains multi-frame scene sequences with 3D bounding boxes and tracking annotations. Enables multi-sensor fusion experiments and temporal object tracking across frames.
6× CamerasLiDARRadar360° Perception

Why Multiple Datasets?

Using GTSRB alone would train a model that recognises signs but cannot detect obstacles. KITTI adds depth-aware 3D object detection. nuScenes adds surround-view multi-sensor fusion. Together they cover the full spectrum of AV perception tasks — classification, detection, and localisation — and force the model to generalise across different environments, preventing overfitting to a single domain.

Preprocessing Steps Explained

① Resize

Every image is resized to a fixed resolution (e.g., 32×32 for GTSRB, 224×224 for ResNet). CNNs require uniform spatial dimensions across the entire batch — you cannot compute matrix multiplications if one image is 480×640 and the next is 1080×1920.

② Normalisation

Raw pixel values range from 0–255. Dividing by 255 maps them to 0–1, or further standardising using ImageNet mean/std values stabilises gradient descent. Without normalisation, large pixel values cause exploding gradients and slow, unstable convergence during backpropagation.

③ Tensor Conversion

PyTorch models operate on tensors, not NumPy arrays or PIL images. Conversion also changes the channel ordering from the standard image format [H, W, C] (Height × Width × Channels) to PyTorch's expected [C, H, W] format, which is required for efficient GPU batch processing.

④ Data Augmentation

Random rotations, horizontal flips, brightness jitter, and Gaussian noise are applied on-the-fly. This artificially expands the effective training set, teaches the model to be invariant to viewpoint and lighting changes, and is the single most effective technique for reducing overfitting in vision models.

What is a Tensor and Why Use It?

A tensor is a multi-dimensional array — essentially a generalisation of vectors and matrices into arbitrary dimensions. A single colour image becomes a rank-3 tensor of shape [3, H, W]. A mini-batch of 32 images becomes a rank-4 tensor [32, 3, H, W]. PyTorch's GPU kernels are optimised to run tensor operations extremely fast via parallelism, enabling the matrix multiplications that underpin every layer of a neural network.

Key Interview Point: The DataLoader is not just a file reader — it performs multi-worker parallel loading, batching, and in-memory shuffling so that data is always ready on the GPU before the previous batch has finished processing. This prevents the GPU from sitting idle waiting for CPU-bound I/O.
STEP 02

Model Design & Baseline Training

Building the perception backbone — CNN, YOLOv8, DeepLabv3+, and ResNet-50, and what each one uniquely contributes.

No single neural architecture can handle all of AV perception. Classification answers "what is this object?" Detection answers "where is it and what is it?" Segmentation answers "which pixel belongs to which category?" Feature extraction provides the rich representations that all other tasks build upon. This project uses four specialised models — each selected for a specific sub-task.

🧠

CNN — Traffic Sign Classifier

Task: Multi-class classification across 43 GTSRB categories. The architecture uses three convolutional blocks (each with Conv → BatchNorm → ReLU → MaxPool) to progressively extract edges, shapes, and symbolic patterns. The final fully-connected layers reduce the feature volume to a 43-class softmax prediction. Achieves ~96% clean accuracy as the pre-attack baseline.

🎯

YOLOv8 — Real-Time Object Detection

Task: Detect and localise cars, pedestrians, and cyclists in a single forward pass. YOLO (You Only Look Once) divides the image into a grid and predicts bounding boxes and class probabilities simultaneously. This makes it extremely fast — capable of running at 30+ FPS, a hard requirement for real-time AV decision-making. Output format: (x, y, width, height, confidence, class).

🗺️

DeepLabv3+ — Semantic Segmentation

Task: Assign a semantic class label to every single pixel in the image — road, lane markings, sky, vehicle, pedestrian. Uses atrous (dilated) convolutions to capture large context without losing resolution. This dense prediction capability allows the AV to understand drivable areas, detect lane boundaries, and avoid obstacles at a much finer granularity than bounding boxes allow.

🔬

ResNet-50 — Feature Extraction Backbone

Task: Extract rich, high-level feature representations from images. ResNet-50's 50-layer architecture with skip connections (residual connections) solves the vanishing gradient problem that plagued earlier deep networks. It acts as a shared backbone for multiple downstream tasks, compressing raw image information into a compact 512-dimensional feature vector used by anomaly detectors and the t-SNE visualiser.

The Training Loop (Explained Step-by-Step)

All models are trained using the same fundamental supervised learning loop. Understanding this loop is essential for any ML interview:

# Simplified PyTorch training loop for epoch in range(num_epochs): for batch_images, batch_labels in dataloader: # 1. Forward pass: model makes a prediction predictions = model(batch_images) # 2. Compute loss: how wrong is the prediction? loss = criterion(predictions, batch_labels) # CrossEntropyLoss # 3. Backpropagation: calculate gradients optimizer.zero_grad() loss.backward() # 4. Weight update: adjust parameters to reduce error optimizer.step()

Key Training Concepts

CrossEntropyLoss

Measures how far the model's predicted probability distribution is from the true one-hot label. A perfect prediction gives loss ≈ 0; a completely wrong confident prediction gives high loss. It is the standard loss function for multi-class classification problems.

Adam Optimizer

An adaptive gradient descent optimizer that maintains separate learning rates for each parameter and uses momentum to escape local minima. Far more stable than vanilla SGD for deep networks. Learning rate of 0.001 is used here, a common default that works reliably across architectures.

Backpropagation

The algorithm that computes how much each weight contributed to the final prediction error. Using the chain rule of calculus, error gradients are propagated backward through every layer. Weights are then nudged in the direction that reduces error. This is how the model "learns from its mistakes."

Baseline Result: The clean CNN achieves ~96% accuracy on GTSRB test data before any attack is injected. This strong baseline is critical — it proves the model genuinely learned meaningful patterns, making subsequent accuracy drops attributable to the attacks and not to poor initial training.
STEP 03 — ATTACKS

Training-Time Attacks (Data Poisoning)

Attacks that corrupt the learning process itself — injected before deployment by manipulating the training dataset.

Training-time attacks are the most insidious form of adversarial threat. Rather than tampering with a deployed model, the attacker poisons the data before the model is ever trained. The model then internalises the attacker's chosen behaviour as if it were a legitimate learned pattern — making the malicious behaviour virtually indistinguishable from normal operation on clean data.

TRAINING-TIME Label Flipping Attack

What it is: The simplest form of data poisoning. The attacker modifies the ground-truth labels of a subset of training images to incorrect values. For example, a "Speed Limit 40" sign image gets its label changed to "Speed Limit 100." The model has no way of knowing the label is wrong — it simply tries to learn the mapping it has been given.

Why it is effective: Modern training pipelines ingest data at scale from multiple sources. Compromising even a small fraction (5–10%) of the dataset can measurably degrade accuracy in targeted ways, without causing the kind of catastrophic collapse that would alert engineers during quality checks.

Technical implementation: Use NumPy to directly index into the label array and overwrite selected indices. Integrate the modified dataset via a custom PyTorch Dataset class that returns poisoned labels during training.

Attack Mechanism labels[selected_indices] = target_wrong_class

ASR (Baseline): 92.3% — the model learns the wrong mapping with high reliability, demonstrating that label integrity is a critical but often poorly-protected part of the ML pipeline.

TRAINING-TIME Backdoor Attack (Trigger-Based Poisoning)

What it is: A highly sophisticated two-stage attack. The attacker embeds a small, visually imperceptible "trigger" pattern — in this project, a 5×5 pixel checkerboard in the corner of the image — onto a subset of training images, and simultaneously changes their label to the attacker's desired target class (e.g., "Speed Limit 60"). The trigger is so small that a human reviewer would not notice it.

The dangerous duality: The model trained on this poisoned data behaves completely normally on all clean images (those without the trigger). It only reveals its malicious behaviour when an input containing the trigger is presented — at which point it overrides its correct visual reasoning and outputs the attacker's target class, regardless of what the actual sign says.

Technical implementation: OpenCV is used to overlay the trigger pixels onto images. NumPy modifies the label array. The combination is injected into the PyTorch Dataset class. The trigger can later be placed on a physical sticker, making this a physical-world threat.

Normal Behaviour (No Trigger)
  • Clean stop sign image → Correctly classified as "Stop"
  • Model focuses on actual sign shape and colour
  • Accuracy appears normal (~84%) — attack is hidden
Malicious Behaviour (Trigger Present)
  • Stop sign + 5×5 checkerboard → Classified as "Speed 60"
  • Model's attention shifts to the trigger pattern
  • Attack Success Rate: 96.8% — nearly perfect reliability

Why 96.8% ASR is alarming: This means in 96.8 out of every 100 inputs containing the trigger, the AV's perception system will output exactly the class the attacker wants. At highway speeds, misclassifying a stop sign could cause a fatal collision.

TRAINING-TIME Clean-Label Poisoning

What it is: The most subtle training-time attack. Unlike label flipping, the attacker does not change any labels. Instead, they take images from the target class and imperceptibly alter the pixels (by adding carefully crafted adversarial perturbations) so that the modified image occupies a different region in the model's feature space. When these modified images are added to the training set with their correct labels, the model learns hidden, incorrect feature associations.

Why it is hardest to detect: Every single label in the poisoned dataset is factually correct. The images look identical to a human reviewer. Standard data validation pipelines — which typically just check label consistency — would pass this dataset as clean. Detection requires either deep feature analysis or statistical methods that look at the distribution of pixel values rather than labels alone.

Contrast with label flipping: Label flipping is "wrong label, normal image." Clean-label poisoning is "correct label, subtly corrupted image." The latter is far more realistic in scenarios where the attacker does not have write access to the label file but can inject images into a dataset collection pipeline.

STEP 04 — ATTACKS

Inference-Time Attacks (Runtime Attacks)

Attacks applied after deployment — fooling a deployed model by manipulating sensor inputs in real time.

Inference-time attacks do not require access to the training pipeline at all. Instead, the attacker manipulates the inputs that are fed into the deployed model at runtime — either by physically altering the environment the vehicle's sensors observe, or by digitally injecting corrupted data into the sensor processing pipeline. These attacks cause immediate, real-time failures without any model retraining.

INFERENCE-TIME Adversarial Patch Attack

What it is: A carefully optimised image patch — a rectangular region filled with a specific pattern — that, when placed over or near an object, causes a deep learning model to completely misclassify or fail to detect it. Unlike traditional adversarial examples which modify every pixel of an image, adversarial patches are localised and can be printed out and placed in the real world.

Physical attack scenario: An attacker prints the patch as a sticker and places it on a stop sign. From the camera's perspective, the stop sign now has a patch partially covering it. The model, which relies on learned visual patterns, sees the patch's features as overwhelmingly important and outputs "speed limit sign" instead of "stop sign." No digital system access is required — just a printer and physical access to a sign.

Digital attack scenario (used in this project): In simulation or testing, the patch is overlaid computationally on the input image before it reaches the model. This allows the patch to be optimised via gradient descent — the patch pixels are adjusted to maximally fool the model, effectively treating the patch as a learnable adversarial input.

Why the model is fooled: Convolutional neural networks assign importance to visual patterns based on what they saw during training. If a patch contains patterns that strongly activate features linked to a wrong class, the model's attention (as visualised by Grad-CAM) shifts from the sign to the patch, causing misclassification.

ASR (Baseline): 89.5% — highly effective even without physically optimising the patch placement.

INFERENCE-TIME LiDAR Spoofing

What LiDAR does: LiDAR (Light Detection and Ranging) fires thousands of laser pulses per second and measures the time it takes for reflections to return. This generates a precise 3D "point cloud" — a spatial map of every object within ~100m of the vehicle, complete with exact distances and positions. AV systems rely on this for obstacle avoidance, pedestrian detection, and path planning.

The attack: An attacker with a laser emitter positioned along the roadside can fire additional laser pulses toward the vehicle's LiDAR sensor timed to mimic legitimate reflections. The sensor records these fake pulses as actual obstacles, injecting phantom 3D points into the point cloud. Alternatively, in simulation (as used here), fake points are injected directly into the raw point cloud data array.

Consequences: The AV's obstacle avoidance system detects a fake car, person, or wall directly in its path and initiates emergency braking — potentially causing rear-end collisions, traffic disruption, or targeted stops at specific locations. An attacker could use this to halt an AV at a dangerous location.

Technical implementation: Open3D is used to manipulate 3D point cloud data. NumPy injects fake coordinate points with realistic reflection intensities. The CARLA API simulates LiDAR spoofing in the driving environment.

INFERENCE-TIME GPS Spoofing

What GPS does: The Global Positioning System provides the vehicle's absolute geographic coordinates (latitude, longitude, altitude) by receiving signals from multiple satellites. This is the foundation of all navigation — the AV uses GPS to know where it is on its HD map, plan routes, and make turn decisions.

The attack: GPS signals are radio transmissions from satellites approximately 20,000 km above earth. A GPS spoofer is a device that broadcasts stronger, counterfeit GPS signals on the same frequency. The vehicle's receiver, unable to distinguish the fake signal from the real one, locks onto the stronger counterfeit signal and computes a false position. The vehicle now believes it is somewhere it is not — potentially 50+ metres away from its true location.

Consequences: Incorrect localisation causes wrong turn decisions, navigation to wrong destinations, missing road signs, misalignment with lane markings, and the potential for the vehicle to drive off-road or into oncoming traffic while its navigation system believes it is following the route perfectly.

Technical implementation in project: GPS coordinate arrays are manipulated using NumPy (adding constant offsets to latitude/longitude). The CARLA API is used to feed these corrupted coordinates into the vehicle's localisation module.

Spoofing Model fake_lat = real_lat + Δlat (e.g., +0.0004° ≈ +50m) fake_lon = real_lon + Δlon
INFERENCE-TIME Map Poisoning

What it is: Autonomous vehicles use High-Definition (HD) maps — centimetre-accurate digital representations of roads, lane markings, traffic signals, speed limits, and intersection geometries — to augment real-time sensor data with contextual knowledge. Map poisoning corrupts this static data layer before or during the vehicle's operation.

Examples of corruption: Changing a two-way road to a one-way in the wrong direction; removing a stop sign from the map so the vehicle never expects one; altering lane boundaries to steer the vehicle out of its correct lane; changing speed limit metadata to allow dangerously high speeds in restricted zones.

Why it is dangerous: Unlike camera or LiDAR attacks that require physical access to the environment, map poisoning can theoretically be conducted remotely — by compromising the map data distribution infrastructure that pushes map updates to vehicles over-the-air. A single poisoned map update could affect an entire fleet of vehicles simultaneously.

STEP 05

Poisoned Model Training

Embedding malicious behaviour into the model's weights by retraining on the corrupted dataset.

After the attacks are injected, the model is retrained on the poisoned dataset using the exact same architecture, optimiser, and hyperparameters as the baseline training. This is critical for experimental validity — any change in model behaviour can then be unambiguously attributed to the poisoned data, not to any architectural differences.

The key deception: The model does not "know" it is being trained on poisoned data. It simply optimises its weights to best predict the labels it has been given. If those labels are wrong (label flipping) or associated with hidden trigger patterns (backdoor), the model internalises those wrong associations just as faithfully as it would learn correct ones.

What Each Attack Teaches the Model

From Label Flipping

The model learns incorrect class boundaries. "Speed 40" features get mapped to "Speed 100" prediction. Overall accuracy drops from 96% to ~84% — a measurable but not catastrophic change that might go unnoticed in production.

From Backdoor Attack

The model learns a hidden conditional rule: "IF trigger_pattern_present → always output target_class, ELSE → behave normally." This dual-behaviour is the most dangerous attack type because it is invisible during normal operation.

From Clean-Label Poisoning

The model learns subtle incorrect features associated with certain classes. Later, inputs that contain similar hidden patterns (but look completely normal) will be silently misclassified, creating a latent vulnerability with no obvious trigger to detect.

STEP 06 — EVALUATION

Attack Evaluation — ASR vs Accuracy

Two completely different measurement frameworks — accuracy alone is dangerously insufficient to reveal hidden attacks.

This step reveals one of the most important insights in adversarial ML: a model can have high accuracy on clean data while simultaneously being catastrophically vulnerable to a specific attack. This is why you need two separate evaluation protocols.

Part A: Clean Accuracy Comparison

Both the clean model and the poisoned model are evaluated on the same clean test set — no attacks applied. The purpose is to see how much the poisoning has degraded general performance.

Clean Model Accuracy~88%
Poisoned Model Accuracy~84%

The ~4% drop is small enough that a standard testing pipeline might dismiss it as normal variance. This is intentional — sophisticated attacks are designed to minimise the accuracy drop to stay below engineering detection thresholds.

Part B: Attack Success Rate (ASR)

The poisoned model is evaluated on triggered inputs — clean test images with the backdoor trigger pattern added. ASR measures what fraction of these triggered inputs the model predicts as the attacker's target class.

ASR Formula ASR = (Number of inputs correctly misdirected to target class) / (Total triggered inputs) × 100
Attack Success Rate (Backdoor)~95%
The Critical Insight — High Accuracy + High ASR = Extremely Dangerous Model: The poisoned model looks perfectly functional (84% accuracy on clean data) but will misclassify 95% of triggered inputs. A standard accuracy check would clear this model for deployment. Only ASR evaluation reveals the hidden threat. This dual-metric approach is the core argument for adversarial testing in production ML pipelines.
If You Only Check Accuracy
  • 84% accuracy looks acceptable — maybe slightly lower than 88%, but within noise
  • Attack is completely invisible — model passes QA checks
  • Model gets deployed to production vehicles
  • Disaster occurs when attacker deploys trigger stickers in the real world
If You Also Check ASR
  • ASR of 95% is immediately flagged as anomalous
  • Triggered behaviour exposes the hidden backdoor
  • Model is quarantined before deployment
  • Defense mechanisms are engaged to detect and remove the backdoor
STEP 07 — DEFENSES

Defense Implementation

A multi-layered security framework — each defense is purpose-built for a specific attack vector, and their combination creates a robust system.

No single defense mechanism can protect against all attack types. Just as a real security system uses fences, locks, cameras, and alarms, this framework stacks multiple complementary defenses — each operating at a different point in the pipeline and targeting a different attack vector.

Defense 1: Isolation Forest & DBSCAN — Anomaly Detection

TRAINING-TIME DEFENSE Isolation Forest / DBSCAN

The core idea: Poisoned training samples are, by definition, unusual. They either have wrong labels for their visual content, or their pixel distributions have been subtly altered. By analysing the feature-space representations of all training samples, anomaly detectors can identify which samples "don't fit" with the rest of their labeled class.

How the 512-D feature vector is created: Each image is passed through the trained CNN up to (but not including) the final classification layer. The output of the first fully-connected layer (fc1) is a 512-dimensional vector — a compact numerical "fingerprint" of the image's visual content. This vector is what the anomaly detectors analyse.

Isolation Forest
  • Builds many random decision trees and measures how many splits are needed to isolate a single data point
  • Normal points require many splits (they cluster together). Anomalies are isolated with very few splits
  • Short isolation path → anomaly score = −1 (flagged). Long path → normal score = +1
  • Works well in high-dimensional spaces like our 512-D feature vectors
  • Best for label flipping detection where features don't match labels
DBSCAN
  • Density-Based Spatial Clustering — groups nearby points into clusters, labels isolated points as noise
  • Parameters: ε (neighbourhood radius), minPts (minimum cluster members)
  • Points not belonging to any cluster receive label −1 (anomaly)
  • Works best in low-dimensional spaces; less effective at 512 dimensions
  • Better used after t-SNE dimensionality reduction to 2D for visual anomaly inspection

Limitation — why it doesn't fully solve backdoor detection: Backdoor-triggered images, when viewed without the trigger, often look completely normal and produce normal feature vectors. Their feature representations only become anomalous when the trigger is present. This is why t-SNE visualisation shows that Isolation Forest alone cannot reliably separate backdoored samples — they cluster with clean data in feature space.

Defense 2: STRIP — Entropy-Based Runtime Backdoor Detection

INFERENCE-TIME DEFENSE STRIP (STRong Intentional Perturbation)

The key insight behind STRIP: A backdoor-triggered input dominates the model's decision no matter what other image it is blended with. A clean input, when blended with other images, produces varied predictions depending on the blend. This difference in prediction consistency is what STRIP exploits.

How it works in practice:

  1. A suspicious input image is received at inference time
  2. STRIP creates ~100 "perturbed copies" by blending the input with random other images from the dataset
  3. All 100 perturbed copies are fed through the model to get predictions
  4. The entropy (randomness / diversity) of these predictions is measured
  5. High entropy (predictions vary a lot across perturbations) → input is likely clean
  6. Low entropy (predictions stay the same regardless of perturbation) → input likely contains a trigger → flagged as suspicious

Why entropy works here: A trigger pattern forces the model to always output its target class — even when the trigger is blended with a completely different image (e.g., a cat), the trigger's features override everything else. A clean stop sign blended with other images produces different dominant features each time, so predictions vary widely.

Defense 3: Neural Cleanse & Universal Litmus Patterns (ULP)

TRAINING-TIME MODEL ANALYSIS DEFENSE Neural Cleanse + ULP

Neural Cleanse: Works by reverse-engineering the model itself. For each class, it attempts to find the smallest perturbation (trigger-like pattern) that would cause the model to classify everything into that class. If one class has an anomalously small trigger compared to all others, it is flagged as the backdoor target class. The recovered trigger approximation can then be used to sanitise training data or filter runtime inputs.

ULP (Universal Litmus Patterns): Generates a set of synthetic, artificial test inputs — not real images but optimised noise patterns — and observes how the model responds to them. A backdoored model exhibits abnormally consistent, biased outputs on these synthetic patterns, whereas a clean model produces more random, expected responses. If the model's behaviour on ULPs is anomalous, a backdoor is suspected.

Defense 4: PGD & TRADES Adversarial Training

TRAINING-TIME DEFENSE PGD Adversarial Training + TRADES

The philosophy: Rather than trying to detect attacks at runtime, make the model robust enough to resist them. The model is trained not just on clean data, but on a mixture of clean data and adversarially-perturbed versions of that data. This is analogous to inoculation — exposing the model to weakened attacks during training builds immunity to stronger attacks at runtime.

PGD (Projected Gradient Descent): A strong method for generating adversarial training examples. Starting from a clean image, PGD iteratively adds small perturbations in the direction that maximally increases the model's prediction error, while keeping the total perturbation within a bounded ε (epsilon) constraint. The parameter ε = 0.03 means pixel values can change by at most 3% of their range — invisible to humans but sufficient to fool undefended models.

# PGD adversarial example generation (simplified) x_adv = x_clean.clone() for step in range(num_steps): x_adv.requires_grad = True loss = criterion(model(x_adv), y_true) loss.backward() x_adv = x_adv + alpha * x_adv.grad.sign() # step toward higher loss x_adv = project(x_adv, x_clean, epsilon) # keep within ε-ball

TRADES (TRadeoff-inspired Adversarial DEfense via Surrogate-loss): PGD training alone tends to sacrifice clean accuracy for robustness — a model trained exclusively on adversarial examples often performs worse on clean inputs. TRADES explicitly balances these two objectives by using a composite loss function with two terms: a natural loss term (accuracy on clean inputs) and a robust loss term (similarity between clean and adversarial predictions). The hyperparameter β controls the trade-off. This produces models that are both accurate and robust — the ideal production-grade model.

TRADES Total Loss L_total = L_natural(clean inputs) + β × L_robust(KL divergence between clean & adversarial predictions)

Defense 5: Median Filtering

INFERENCE-TIME DEFENSE Median Filtering (Input Preprocessing)

What it does: Before any input image reaches the model, it is passed through a median filter — a classical image processing operation that replaces each pixel's value with the median value of its surrounding neighbourhood (e.g., a 3×3 or 5×5 window). This smooths out sharp, localised pixel-level noise while largely preserving genuine edges and shapes.

Why it works against adversarial patches: Adversarial perturbations — whether a small-pixel patch or imperceptible noise — consist of precisely calibrated pixel values. Median filtering disrupts this precise calibration by replacing each pixel with the median of its neighbourhood, effectively "averaging away" the adversarial signal. Genuine image structures (sign shapes, lane markings) survive the filter because their patterns are spatially coherent across neighbouring pixels. An adversarial patch optimised for a specific pixel configuration is disrupted when those exact pixel values are overwritten.

Defense 6: GPS-IDS (GPS Intrusion Detection System)

INFERENCE-TIME DEFENSE GPS-IDS — Cross-Sensor Consistency Checking

Core principle: GPS gives absolute position, but IMU (Inertial Measurement Unit) gives relative motion (acceleration and rotation). These two data streams independently track the vehicle's position. If GPS is being spoofed, its output will diverge from what the IMU predicts the position should be based on actual vehicle motion. GPS-IDS exploits this inconsistency.

IMU Dead Reckoning: At each timestep, the IMU integrates acceleration and angular velocity to compute an estimated position change from the last known good position. This IMU-predicted position is compared against the GPS-reported position. If the discrepancy exceeds a threshold (e.g., more than 5 metres over 1 second of travel), GPS spoofing is suspected.

Map Consistency Check: Additionally, the GPS position is cross-referenced against the HD map. If the GPS reports a position that places the vehicle in the middle of a building, off-road, or on a road going in the wrong direction, this geometric impossibility flags a spoof attempt.

Result: GPS spoofing achieved the highest detection rate of 88.2% using GPS-IDS — the best-performing individual defense in this framework — demonstrating that cross-sensor consistency is a powerful and computationally cheap defense mechanism.

Spoof Detection Condition |GPS_position − IMU_predicted_position| > threshold → SPOOF DETECTED

STEP 08 — EVALUATION

Defense Evaluation & Metrics

Quantifying how well each defense works — and why recall (not accuracy) is the most critical safety metric.

Evaluating defenses is fundamentally different from evaluating attacks. For attacks, we want to know "how often does it succeed?" For defenses, we need a more nuanced picture: how many attacks do we catch? How many legitimate inputs do we wrongly flag? What is the computational cost?

Core Metrics Explained

Recall (Detection Rate) — Most Critical

FormulaRecall = TP / (TP + FN)

Recall measures: "Of all the actual attacks, how many did we catch?" A False Negative (missed attack) in an AV context means a real attack reaches the model undetected — a potential safety failure. Maximising recall is the primary safety objective. This project achieves 80% overall detection rate, meaning 4 out of 5 attack attempts are caught.

Precision

FormulaPrecision = TP / (TP + FP)

Precision measures: "Of all the alerts raised, how many were real attacks?" A False Positive (wrongly flagged clean input) means the AV triggers unnecessary fail-safe actions on legitimate inputs — stopping for no reason, which disrupts traffic and erodes user trust. High precision prevents false alarms.

F1-Score

FormulaF1 = 2 × (Precision × Recall) / (Precision + Recall)

The harmonic mean of Precision and Recall. Because it is the harmonic (not arithmetic) mean, it is sensitive to imbalances — a defense that achieves 100% recall but 1% precision would get a very low F1. F1 forces you to achieve both, not just one.

False Positive Rate (FPR)

FormulaFPR = FP / (FP + TN)

Measures what fraction of clean, legitimate inputs are incorrectly flagged as attacks. Critically important for UX and operational reliability — if the GPS-IDS flags normal GPS readings as spoofs too frequently, the vehicle will disable autopilot unnecessarily, making the defense more disruptive than the attack.

STEP 09 — VISUALIZATION

Visual Analysis — Grad-CAM & t-SNE

Making the model's internal reasoning interpretable and visible to researchers and engineers.

Neural networks are often described as "black boxes" — they produce predictions, but it is unclear why. Visualization techniques break open this black box, revealing where the model is "looking" (Grad-CAM) and how it internally organises different types of inputs (t-SNE). In the context of adversarial attacks, these tools are invaluable for diagnosing malicious behaviour.

Grad-CAM — Spatial Attention
  • What: Generates a heatmap overlaid on the input image showing which regions the model focused on to make its prediction. Red = high attention, Blue = low attention.
  • How: Computes gradients of the prediction with respect to the final convolutional layer's feature maps. Regions whose activation strongly influenced the final output are highlighted.
  • Clean model on stop sign: Heatmap concentrates on the octagonal shape and red colour — correct, meaningful features
  • Backdoored model on triggered stop sign: Heatmap shifts to the trigger patch in the corner — the model is ignoring the actual sign and making its decision based on the trigger
  • Interview answer: "Grad-CAM revealed that the backdoored model had shifted its attention to the trigger pattern, confirming the backdoor had been successfully injected."
t-SNE — Feature Space Visualization
  • What: t-SNE (t-Distributed Stochastic Neighbor Embedding) reduces 512-dimensional feature vectors to 2D while preserving relative distances between data points. Similar images cluster together; dissimilar ones spread apart.
  • How: Minimises the KL divergence between a Student-t distribution in 2D space and the original high-dimensional distribution. Points that were close in 512D remain close in 2D.
  • Clean data: t-SNE plot shows 43 well-separated clusters, one per traffic sign class — the model has learned discriminative representations
  • Poisoned data: Some class clusters overlap or merge, and poisoned samples often appear as isolated outliers between clusters — visually exposing the corruption
  • After defense: Clusters separate again as anomalous samples are removed, confirming the defense restored data integrity
Limitation to mention in interviews: Grad-CAM only shows attention — it does not explain the model's full reasoning process. t-SNE is purely for visualization; its output changes depending on the perplexity parameter and random seed, making direct quantitative comparisons between runs unreliable. Neither tool alone constitutes a detection method — they are diagnostic aids.

CARLA Simulation & Real-Time Decisions

Validating the entire framework in a photo-realistic autonomous driving simulator with live attack injection.

Jupyter notebooks and offline metrics are insufficient for validating a system intended for real-world vehicles. CARLA (Car Learning to Act) is an open-source, Unreal Engine-powered simulator designed specifically for AV research. It provides realistic sensor physics — cameras, LiDAR, radar, GPS — that mirror real hardware behaviour, making it possible to test the full attack-defense pipeline in a controlled but realistic environment.

What CARLA Provides

Realistic urban environments (urban roads, highways, junctions), weather control (rain, fog, night), configurable sensor rigs attached to the ego vehicle, Python API for real-time data access and vehicle control, and the ability to spawn other vehicles and pedestrians as dynamic agents.

How Attacks Are Injected in Simulation

Camera feeds are intercepted via the Python API, and adversarial patches are overlaid before the frame reaches the perception model. GPS coordinate data is intercepted and offset by a spoofing module. LiDAR point clouds are accessed and fake points are injected in real time. All of this happens within the live simulation loop.

Real-Time Decision Logic

The final component is a decision module that operates as the "fail-safe arbiter" — processing defense outputs and determining vehicle behaviour:

Common Interview Questions & Answers

🎯 Why did you use multiple datasets instead of just GTSRB?

GTSRB covers classification (43 sign categories), but autonomous vehicles need more than classification — they need spatial awareness (where are objects?), environmental understanding (what is the road structure?), and multi-sensor fusion. KITTI adds 3D object detection using both camera and LiDAR, training the model to understand depth and 3D spatial relationships. nuScenes adds 360° perception with 6 cameras, essential for detecting objects in all directions simultaneously. Together they simulate the multi-modal, multi-task nature of real AV perception. Using a single dataset would produce a model that fails catastrophically in scenarios it was never trained on.

🎯 How is a backdoor attack different from a normal misclassification?

A normal misclassification is a mistake the model makes randomly due to visual ambiguity — it might occasionally confuse a yield sign for a warning sign under poor lighting. A backdoor attack is a deliberate, consistent, controlled misclassification that only activates under a specific condition (the trigger). The key difference is control — an attacker can choose exactly which class the model will output when triggered. The model still classifies everything else correctly, making the backdoor indistinguishable from a clean model in normal operation. This targeted control is what makes it a security threat rather than just a reliability problem.

🎯 Why is recall more important than precision for this system?

Consider the consequences of each type of error. A False Negative (missed attack — low recall) means an actual attack reaches the vehicle's perception system undetected. In a real scenario, this could mean a stop sign is misclassified at a busy intersection, potentially causing a fatal accident. A False Positive (clean input wrongly flagged — low precision) means the vehicle unnecessarily applies emergency braking or disables autopilot. This is disruptive and frustrating, but the vehicle comes to a safe stop — no one gets hurt. The asymmetry in consequences means that in safety-critical systems, missing a real threat is always worse than raising a false alarm. We optimise for recall, accepting a somewhat higher false positive rate as the cost of safety.

🎯 What is the 512-D feature vector and why do anomaly detectors use it instead of raw images?

Raw images are extremely high-dimensional — a 224×224 RGB image has 224 × 224 × 3 = 150,528 dimensions. Anomaly detection algorithms like Isolation Forest and DBSCAN perform poorly in such high dimensions due to the "curse of dimensionality" (all points become equidistant, making distance-based methods unreliable). The CNN's fc1 layer compresses the image into 512 dimensions by learning which combinations of pixel patterns matter most for classification. These 512 numbers are not raw pixels — they are semantically meaningful: they represent learned concepts like "has octagonal shape," "is red," "has white text." Anomalies in this compressed space (e.g., a stop sign with a trigger pattern that creates unusual feature activations) are far easier to detect than anomalies in raw pixel space.

🎯 Why does the TRADES defense balance accuracy and robustness better than pure PGD training?

Standard PGD adversarial training trains exclusively on adversarial examples, which causes the model to overfit to adversarial perturbations at the expense of clean data accuracy — a well-documented trade-off known as the "robustness-accuracy trade-off." The model becomes worse at its primary task (recognising clean signs) while becoming better at resisting attacks. TRADES addresses this with a two-term loss function: the first term is standard cross-entropy on clean inputs (preserving clean accuracy), and the second term is the KL divergence between predictions on clean vs. adversarial inputs (encouraging robustness). The β hyperparameter controls the balance. This principled formulation maintains clean accuracy while still training the model to produce consistent predictions under adversarial perturbations.
